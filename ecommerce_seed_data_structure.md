# 이커머스 데이터베이스 스키마 및 초기 시드(Seed) 데이터 구조 가이드

이 문서는 제공된 `init.sql`의 스키마 구조와 `DataSeeder.java`의 데이터 생성 로직을 바탕으로, 데이터베이스 내부에 어떤 형태의 초기 데이터가 삽입되어 있는지 파악하기 위해 작성되었습니다. 각 테이블별 스키마와 저장되는 더미(Dummy) 데이터의 실제 형태를 레이어 순서대로 정리했습니다.

---

## 🏗️ 전체 구조 개요
외래 키(FK) 참조 무결성을 보장하기 위해 0단계(최상위)부터 6단계(최하위)까지 순차적으로 생성 및 삽입됩니다.
초기 세팅은 **소규모 테스트 모드**로 구성되어 있습니다. (유저 100명, 상품 200개, 주문 500건 등)

---

## Layer 0: 의존성 없는 독립 테이블 (최상위)

### 1. `users` (회원)
* **총 데이터 수:** 100건 
* **구조 및 데이터 예시:**
  * `email`: `user1@example.com` 형태 (ID 기반으로 중복 회피)
  * `password`: `hashed_password_0`, `hashed_password_1` 형태
  * `name`: `김철수`, `이영희` (Faker 한국 이름 생성)
  * `phone`: `010-1234-5678` (Faker 휴대폰 번호 포맷)
  * `gender`: `MALE`(48%), `FEMALE`(48%), `OTHER`(4%) 비율
  * `birth_date`: 현재 기준 20~60세 사이 (예: `1985-04-15`)
  * `grade`: `BRONZE`(60%), `SILVER`(25%), `GOLD`(10%), `VIP`(5%)
  * `point_balance`: `0` ~ `99,999` 사이 랜덤
  * `created_at` / `updated_at`: 최근 3년 이내 랜덤 시점

### 2. `category` (카테고리)
* **총 데이터 수:** 20건
* **구조 및 데이터 예시:**
  * **대분류(5건, depth=0):** `전자제품`, `패션`, `식품`, `스포츠`, `뷰티` (`parent_id` = null)
  * **중소분류(15건, depth=1):** `뷰티용품`, `컴퓨터/주변기기` (Faker 상거래 부서명 생성기 활용, parent_id는 대분류 ID 중 랜덤 지정)

### 3. `coupon` (쿠폰 마스터)
* **총 데이터 수:** 10건
* **구조 및 데이터 예시:**
  * `name`: `PROMO-ABCD` (Faker 프로모션 코드)
  * `discount_type`: `RATE`(정률, 약 10~40%) 또는 `FIXED`(정액, 약 1000~5000원)
  * `min_order_amount`: `0`, `10000`, `20000` 중 하나
  * `started_at` / `expired_at`: 과거 30일 전 ~ 향후 60일 뒤까지 유효
  * `max_issue_count`: `1000` 고정
  * `issued_count`: `0` ~ `799` 사이 랜덤 (현재까지 발급된 매수)

## Layer 1: Layer 0에 의존

### 4. `user_address` (사용자 배송지)
* **총 데이터 수:** 약 150건 (회원당 평균 1.5개)
* **데이터 형태:** 
  * `address` / `detail_address`: (Faker 한국 도로명 주소 & 상세주소) 
  * `is_default`: 회원이 갖는 첫 번째 주소지는 무조건 기본 주소지(`true`)
  * `receiver_name` / `receiver_phone`: 수령인 실명과 연락처 데이터

### 5. `user_coupon` (사용자 발급 쿠폰)
* **총 데이터 수:** 약 60건 (전체 회원의 60%에게 1장씩 랜덤 발급)
* **데이터 형태:**
  * `is_used`: 약 30% 확률로 이미 사용됨(`true`)
  * `used_at`: 사용된 쿠폰의 경우 최근 30일 이내의 날짜

### 6. `product` (상품 마스터)
* **총 데이터 수:** 200건
* **데이터 형태:**
  * `name`: 랜던 상품명 (예: "편안한 러닝화") 
  * `description`: 랜덤 상품 설명 (라틴어 더미 문장)
  * `base_price`: `10,000`원 ~ `100,000`원 (만원 단위)
  * `status`: `ON_SALE`(80%), `SOLD_OUT`(15%), `DISCONTINUED`(5%)
  * `is_deleted`: 5% 확률로 삭제 상태(`true`)
  * `created_at`: 최근 2년 이내

### 7. `point_history` (포인트 내역)
* **총 데이터 수:** 1,000건
* **데이터 형태:**
  * `type`: `EARN`(60%), `USE`(35%), `EXPIRE`(5%)
  * `amount`: `100` ~ `1000` 사이 (백원 단위)
  * `description`: "포인트 적립", "포인트 사용"
  * `created_at`: 최근 1년 이내 무작위 발생

---

## Layer 2: Layer 1에 의존

### 8. `product_option` (상품 옵션명)
* **총 데이터 수:** 300건 (상품당 약 1.5개)
* **데이터 형태:** `option_name`은 `"색상"`, `"사이즈"`, `"용량"`, `"재질"` 중 하나로 고정.

### 9. `product_image` (상품 갤러리/이미지)
* **총 데이터 수:** 400건 (상품당 2개씩)
* **데이터 형태:** `image_url`은 형태소만 가진 UUID 형태(`https://example.com/images/...jpg`). 짝수 번째 사진이 썸네일(`is_main=true`).

### 10. `product_sku` (상품 SKU / 재고 단위)
* **총 데이터 수:** 600건 (상품당 3개씩)
* **데이터 형태:**
  * `sku_code`: `SKU-1`, `SKU-2` 형태 (절대 중복 방지)
  * `stock_quantity`: `0` ~ `499`개
  * `extra_price`: 기본가에 붙는 추가 금액 `0` ~ `4,000`원 (천원 단위)

### 11. `cart` (장바구니 리스트)
* **총 데이터 수:** 약 70건 (회원의 70%가 보유)
* **데이터 형태:** `created_at` 최근 90일 내 부여. (아직 결제하지 않은 상태를 의미)

---

## Layer 3: Layer 2에 의존

### 12. `product_option_value` (상품 옵션의 구체적 속성값)
* **총 데이터 수:** 900건 (각 옵션당 3개의 속성 부여)
* **데이터 형태:** 옵션이 '색상'일 경우 `빨강`, `파랑`, `초록`. '사이즈'일 경우 `S`, `M`, `L` 등 고정된 배열에서 생성.

### 13. `product_sku_option` (SKU 맵핑 테이블)
* **총 데이터 수:** 600건
* **데이터 형태:** 하나의 SKU마다 1개의 구체적 `옵션 속성값` 식별자가 엮여 있음.

### 14. `cart_item` (장바구니 내역)
* **총 데이터 수:** 140건 (장바구니당 약 2개의 상품 담음)
* **데이터 형태:** 담은 수량 `quantity`는 `1` ~ `3`개로 무작위 지정. 담은 일자는 최근 30일 이내.

### 15. `orders` (주문 마스터)
* **총 데이터 수:** 500건
* **데이터 형태:**
  * `total_price`: `10,000` ~ `100,000`원 내외
  * `discount_price`: 30% 확률로 쿠폰을 사용하며, 이 경우 주문 총액의 10%만큼 할인됨
  * `status`: `DELIVERED`(50%), `SHIPPED`(15%), `PREPARING`(10%), `PAID`(10%), `PENDING`(10%), `CANCELLED`(5%)

---

## Layer 4: Layer 3에 의존

### 16. `order_item` (주문된 상품 내역)
* **총 데이터 수:** 1000건 (주문 1건당 약 2개의 종류 상품 포함)
* **데이터 형태:**
  * `product_name`: 주문 당시의 상품명 스냅샷 (Faker 상품명)
  * `option_info`: `"색상: 빨강"` 등 옵션 조합의 텍스트 형태 보존
  * `unit_price`, `quantity`: 1~3개 구매, 만원~오만원 등 주문 당시 가격 격리보존
  * `status`: 시더 구조상 일괄 `"DELIVERED"` 입력 

### 17. `payment` (결제 트랜잭션 내역)
* **총 데이터 수:** 500건 (주문 1건당 1개매칭)
* **데이터 형태:**
  * `method`: `CARD`, `KAKAO_PAY`, `NAVER_PAY`
  * `status`: `COMPLETED`(80%), `PENDING`, `FAILED`, `REFUNDED`
  * `pg_transaction_id`: `"PG-XXXXXXXX"` 형태 (외부 PG사 고유번호 가정)

### 18. `delivery` (배송 원장)
* **총 데이터 수:** 500건 (주문 1건당 1개 매칭)
* **데이터 형태:**
  * `status`: `DELIVERED`(50%), `SHIPPED`(20%), `DELIVERING`(15%), `PREPARING`(15%)
  * `tracking_number`: `"TRACK-XXXXXXX"` 형태의 랜덤 숫자로 부여

---

## Layer 5: Layer 4에 의존

### 19. `refund` (환불 내역)
* **총 데이터 수:** 약 25건 (전체 결제건의 약 5%)
* **데이터 형태:**
  * `amount`: `5,000` ~ `25,000`원 부분/전체 랜덤 환불
  * `reason`: 임의 생성된 환불 사유 (라틴어 더미 문장)

### 20. `delivery_tracking` (배송 상태별 추적 이력)
* **총 데이터 수:** 2000건 (배송 1건당 여러 개의 진행 상태이력)
* **데이터 형태:** 시간 순서에 따라 `서울 물류센터`, `배송 완료` 등 랜덤 지역 위치(`location`)와 `status` 정보 기록

### 21. `review` (마이페이지 리뷰 내역)
* **총 데이터 수:** 약 300건 (전체 주문 상품의 약 30%만 리뷰 작성)
* **데이터 형태:**
  * `rating`: `1` ~ `5`점 (별점)
  * `content`: 장문의 더미 리뷰 텍스트

---

## Layer 6: Layer 5에 의존 (최하위)

### 22. `review_image` (리뷰 첨부 이미지)
* **총 데이터 수:** 약 120건 (작성된 리뷰의 약 40%만 첨부파일 존재)
* **데이터 형태:** `"https://example.com/review-images/...jpg"`

### 23. `review_like` (리뷰 다른회원 '좋아요' 내역)
* **총 데이터 수:** 약 600건
* **데이터 형태:** 각 리뷰마다 랜덤 사용자(본인 제외 상관없음) 0~2명이 좋아요 누른 이력 (복합 Key `review_id`-`user_id` 중복 방지 처리됨)
